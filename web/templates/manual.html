Getting Started

GETTING STARTED

Login

Navigate to http://localhost:2600. Six operator accounts are available, each with a different role and access level:

  • commander / amos_op1 — CDR Mitchell. Full access to all modules, HAL approval, ROE, admin, System Scripts, Edition Dashboard
  • pilot / wings2026 — CPT Torres. Air domain focus — C2, EW, SIGINT, HAL, AWACS, Sensor Fusion
  • grunt / hooah2026 — SGT Reeves. Ground ops — C2, Countermeasures, HAL, field view
  • sailor / anchor2026 — PO1 Chen. Maritime ops — C2, SIGINT, HAL, field view
  • observer / watch2026 — Analyst Kim. Read-only intel — C2, EW, SIGINT, Cyber, AWACS, Red Force, Cognitive Engine
  • field / tactical2026 — SPC Davis. Tactical field view + voice commands + countermeasures (redirects to Field 3D on login)

Navigation

Click the ☰ MENU button in the top bar to access the full module grid. The top bar also shows:

  • Theater selector — Switch between operational theaters (map flies to selected location)
  • Operator pips — Colored dots showing online operators
  • 💬 Chat — Team chat with 4 channels (general, command, intel, fires)
  • 🎤 Voice — Voice commands (Chrome only)
  • LIVE — Real-time simulation status

Simulation

AMOS runs a real-time simulation with autonomous assets, threats, and environmental effects. The simulation speed can be adjusted from the C2 Console (0.5x to 10x). Assets move, threats patrol, weather changes, and cyber events generate automatically.

THEATERS & LOCATIONS

AMOS supports multiple operational theaters, each with its own force composition:

  • FOB Tehran — Primary theater. Full 25-asset platoon (air/ground/maritime). 10 threats including hostile drones, GPS jammers, vessels, and cyber.
  • MacDill AFB (Tampa) — Balanced force. Air, ground, and maritime assets. Training/CONUS scenario.
  • South China Sea — Maritime-heavy. Double maritime assets, no ground forces. FONOPS/carrier ops scenario.
  • Eastern Europe — Ground-heavy. Double ground forces, no maritime. NATO Eastern Flank combined arms.
  • Test Base — Air + ground only. Development/testing scenario.

Switch theaters via the dropdown in the top bar. The map will fly to the selected location. All assets across all theaters are always live in the simulation.

Adding Locations

Go to Settings → Locations to add custom theaters with lat/lng coordinates and AO boundaries.

Operations

C2 CONSOLE (Home)

The main command interface. Shows the tactical map with all assets (colored by domain) and threats.

Map Controls

  • 📍 WP MODE — Click map to set waypoints for the selected asset
  • CLR WP — Clear waypoints
  • Overlays — Toggle threat heatmap, engagement zones, sensor coverage, domain sectors, weather

C2 Command Panel

Select assets (filter by domain), choose an action (formations, commands, swarm ops), and click ▶ EXEC.

  • Formations — LINE, COLUMN, WEDGE, DIAMOND, SPREAD, ORBIT
  • Commands — HOLD, RTB (return to base), PATROL
  • Swarm — CREATE (new swarm group), CLEAR (dissolve)

DIGITAL TWIN

Full platoon roster with real-time status: battery, comms, fuel, ammo, speed, and domain breakdown chart. Use this for fleet-level situational awareness.

TACTICAL VIEW

Full-screen map optimized for mobile/tablet. Bottom drawer shows asset/threat counts, risk level, alerts, and pending HAL approvals with one-tap APPROVE/DENY buttons.

FIELD 3D VIEW

CesiumJS-based 3D globe view. Click any asset or threat in the side panels to fly the camera to it. Filter by domain (ALL/AIR/GROUND/MARITIME). Mesh links show nearby asset connections.

Intelligence

ELECTRONIC WARFARE

Monitor and control EW operations. View EW-capable assets, active jamming operations, and intercepts.

  • Jam — Select a jammer asset, target frequency, and technique (barrage/spot/sweep)
  • Stop — Cease specific jamming operations
  • Intercepts are automatically generated by EW-capable sensors

SIGINT

Signals intelligence dashboard. Shows intercept log, emitter map, emitter database. Intercepts are classified as HOSTILE, SUSPECT, or UNKNOWN based on frequency analysis.

CYBER OPS

Network topology visualization, kill chain analysis, and active threat management.

  • Topology — Interactive network graph showing HQ, mesh, control, and sensor nodes
  • Kill Chain — Maps events to RECON → WEAPONIZE → DELIVER → EXPLOIT → C2 → EXECUTE stages
  • Block — Block IPs or specific threat events
Combat

COUNTERMEASURES

Engage threats with available assets. Subject to ROE compliance — the system will block engagements that violate current rules of engagement. Successful engagements auto-generate BDA reports.

HAL AUTONOMY ENGINE

The cognitive engine generates recommendations for the commander based on the current tactical picture. Each recommendation shows the asset, target, confidence level, and reasoning.

  • APPROVE — Execute the recommended action
  • DENY — Reject the recommendation
  • Pending recommendations also appear in the Tactical View bottom drawer

RULES OF ENGAGEMENT

Manage ROE posture and custom rules. Commander only.

  • WEAPONS HOLD — Fire only in self-defense
  • WEAPONS TIGHT — Fire only at positively identified threats
  • WEAPONS FREE — Fire at any target not identified as friendly

Custom rules can enforce range limits, domain restrictions, and time windows. All violations are logged.

KILL WEB

F2T2EA kill chain pipeline: Find → Fix → Track → Target → Engage → Assess. Visualizes the sensor-to-shooter chain with human approval gates at the ENGAGE stage.

BATTLE DAMAGE ASSESSMENT

Submit and review BDA reports. Auto-generated from countermeasure engagements. Analytics show effectiveness by weapon type, damage level, and confidence distribution.

MONTE CARLO WARGAMING

Run 1000+ iteration simulations on current force posture. Compares 5 approach strategies × 3 tempos with Markov chain force attrition modeling. Results show win probability, expected casualties, and risk assessment.

CQB / Indoor Operations

CQB OPERATIONS CONSOLE

Close Quarter Battle planning and execution interface for indoor / urban operations. Access via ☰ MENU → CQB Ops.

Building Selector

Choose a building from loaded floor-plans (JSON configs in config/buildings/). The console shows all floors, rooms, doors, windows, and entry points.

Mission Planner

  • Mission Type — BREACH_AND_CLEAR, HOSTAGE_RESCUE, ROOM_CLEAR, EXTRACT_HVT
  • Priority Rooms — Select rooms that must be cleared first
  • Asset Assignment — Assign bipedal assets to the mission from the roster
  • Create Mission — Generates the CQB task plan and pushes it to the executor

Live Status

Shows real-time room clearance progress, asset positions, threat detections, and mission phase (PLANNING → INFILTRATION → EXECUTION → CONSOLIDATION).

Threat Intel Overlay

Overlays hostile positions, IED locations, hostage positions, and civilian contacts from the building's threat-intel data and live perception fusion.

3D TACTICAL VIEWER

WebGL-rendered 3D floor-plan viewer for multi-story buildings. Access via ☰ MENU → 3D Tactical or from the CQB console's 3D VIEW button.

Controls

  • Click + Drag — Orbit the camera around the building
  • Scroll — Zoom in/out
  • Right-Click + Drag — Pan the view
  • Floor slider — Isolate individual floors or show all

Layers

  • Rooms — Color-coded by clearance status (grey = unknown, green = cleared, red = hostile, blue = friendly occupied)
  • Assets — Bipedal unit positions rendered as markers with callsign labels
  • Threats — Detected hostiles, IEDs, and unknowns rendered with threat icons
  • SLAM Grid — Occupancy grid overlay showing explored vs. unexplored areas

Requires WebGL 2.0 — tested in Chrome, Edge, and Firefox.

BUILDING MODEL

Manages structural data for indoor environments. Buildings are loaded from JSON files in config/buildings/.

Data Structure

  • Building — ID, name, type, location (lat/lng), floor count, dimensions
  • Floors — Floor number, label, elevation, ceiling height
  • Rooms — ID, name, type (office, corridor, stairwell, etc.), polygon vertices, area
  • Doors — Connects two rooms or room-to-exterior, type (standard, reinforced, emergency), locked status
  • Windows — Room association, wall position, size, breakable flag
  • Entry Points — Named exterior access points with coordinates
  • Threat Intel — Pre-populated hostile positions, IED locations, hostage locations

Adding Buildings

Create a JSON file in config/buildings/ following the schema of existing files (e.g. alpha_compound.json, embassy_bravo.json). Buildings are loaded on startup.

INDOOR POSITIONING

Tracks asset positions inside buildings with sub-room accuracy. Supports multiple position sources fused by confidence weighting.

Position Sources

  • UWB — Ultra-wideband beacons (highest accuracy, ~10 cm)
  • SLAM — Simultaneous localization and mapping from onboard sensors
  • IMU — Inertial measurement unit dead reckoning (lowest accuracy, drift-prone)
  • MANUAL — Operator-reported positions

API

  • GET /api/v1/indoor/positions — All tracked indoor positions
  • GET /api/v1/indoor/positions/<building_id> — Positions within a specific building
  • POST /api/v1/indoor/position — Update an asset's indoor position

Positions include building ID, floor, room, x/y/z coordinates (metres), source type, confidence (0–1), and timestamp.

PERCEPTION FUSION

Fuses detections from multiple assets into a unified indoor threat picture. Combines visual, thermal, LIDAR, and acoustic sensor data.

Detection Pipeline

  • Ingest — Raw detections from individual assets (classification, confidence, position)
  • Correlate — Matches detections within a configurable radius to existing tracks
  • Track — Maintains persistent tracks with smoothed positions and aggregated confidence
  • Classify — Final classification: HOSTILE, CIVILIAN, UNKNOWN, IED, FURNITURE

SLAM Grid

Aggregates occupancy grid scans from assets performing SLAM. Each cell is classified as FREE, OCCUPIED, or UNKNOWN. Coverage percentage is tracked per floor.

API

  • GET /api/v1/perception/tracks — All fused tracks
  • GET /api/v1/perception/tracks/<building_id> — Tracks in a building
  • GET /api/v1/perception/slam/<building_id>/<floor> — SLAM grid for a floor
  • POST /api/v1/perception/detect — Submit a raw detection

SQUAD SUPERVISOR

Manages bipedal squad composition, readiness, and formation assignments for CQB missions.

Squad Management

  • Create Squad — Group bipedal assets into a named squad with a designated leader
  • Readiness — Tracks each member's status (READY, ENGAGED, DOWN, OFFLINE)
  • Formation — Assign tactical formations (STACK, LINE, WEDGE, DIAMOND, ECHELON, BOUNDING_OVERWATCH)

CQB Formations

Squad formations are optimized for indoor environments:

  • STACK — Single-file entry formation for doorway breaches
  • LINE — Abreast for corridor sweeps
  • WEDGE — Point-forward for open rooms
  • DIAMOND — 360° security for large spaces
  • ECHELON — Angled advance for hallway intersections
  • BOUNDING_OVERWATCH — Alternating movement with cover

API

  • GET /api/v1/squad/list — All squads with members and status
  • POST /api/v1/squad/create — Create a new squad
  • POST /api/v1/squad/<id>/formation — Set squad formation

DimOS BRIDGE

Integration bridge to the DimOS (Dimensionality OS) robotic autonomy framework. Enables AMOS to command real or simulated bipedal robots.

Capabilities

  • Navigation — Send waypoint goals to robot navigation stack
  • Manipulation — Door opening, object retrieval, breach actions
  • Perception — Receive sensor feeds (RGB, depth, LIDAR, thermal)
  • Status — Battery, joint states, locomotion mode, error codes

Connection

Configure the DimOS endpoint in .env via DIMOS_ENDPOINT. The bridge maintains a persistent WebSocket connection with automatic reconnect. Connection status is shown on the CQB console.

When no DimOS endpoint is configured, the bridge operates in simulation mode using AMOS's internal physics.

CQB TASK LANGUAGE

The CQB planner generates structured task plans using a domain-specific task language. Tasks are hierarchical and executable by the CQB executor.

Task Types

  • MOVE_TO_ROOM — Navigate an asset to a target room via shortest door path
  • BREACH_DOOR — Breach a locked/reinforced door (explosive, mechanical, or manual)
  • CLEAR_ROOM — Systematic room clearance with threat engagement ROE
  • HOLD_POSITION — Maintain position and provide security overwatch
  • EXTRACT_PERSON — Locate and extract a hostage or HVT from a room
  • STACK_FORMATION — Form up in stack at a door before breach
  • SCAN_ROOM — Perform a SLAM/perception scan without entry

Execution

Tasks are executed sequentially by the CQB Executor, which tracks state per task (PENDING → ACTIVE → COMPLETE / FAILED). Failed tasks trigger automatic re-planning if the mission allows it.

Autonomy

SWARM INTELLIGENCE

Reynolds flocking physics with behavioral DNA profiles. 5 base behaviors + 6 emergent behaviors (surround, funnel, pincer, screen, relay, decoy). Create swarms from the C2 Console or Mission Planner.

ISR/ATR PIPELINE

Automatic Target Recognition with multi-sensor confidence scoring. Pattern-of-Life analysis tracks 8 states (STATIC through TRANSITING). Change detection triggers alerts.

EFFECTS CHAIN

Cross-domain effects orchestration: Cyber → EW → SIGINT → Kinetic → ISR. 5 templates: SEAD, CYBER_KINETIC, QUICK_STRIKE, FULL_SPECTRUM, EW_CORRIDOR. Auto-replans on stage failure.

SPACE / JADC2

9 orbital assets (GPS, SATCOM, ISR, early warning, SIGINT, relay). Keplerian orbital propagation, SATCOM link budgets, GPS denial zones. JADC2 mesh connects space ↔ air ↔ ground ↔ maritime.

HUMAN-MACHINE TEAMING

5 autonomy levels: Manual → Advisory → Consensual → Monitored → Autonomous. Trust calibration per domain, workload classification, fatigue assessment. The system adapts autonomy based on operator state.

MESH NETWORK

MANET simulation with 7 frequency bands. Dijkstra routing, frequency hopping (32-element sequences), store-and-forward queuing. Priority classes: FLASH / IMMEDIATE / PRIORITY / ROUTINE. Network resilience scored A–F.

Support

LOGISTICS & SUPPLY

Per-asset supply tracking: fuel, ammo, water, rations. Color-coded GREEN/AMBER/RED status. Resupply assets via the interface. Supply consumption history tracks usage over time.

WEATHER

Real-time weather conditions with domain-specific impact scores (air/ground/maritime/cyber). Weather overlay on the map shows localized conditions. Mission recommendation: GO / CAUTION / HOLD.

THREAT PREDICTIONS

AI-powered threat trajectory prediction with confidence intervals. Movement pattern detection (STATIC, CIRCLING, LINEAR_APPROACH, EVASIVE). Intercept window calculation with feasibility rating.

COMMS NETWORK

Visualize communication links between all assets. Shows link quality, bandwidth usage, and mesh topology. Identifies disconnected nodes and single points of failure.

AWACS

Airborne command drone management. Shows AWACS assets, radar coverage circles (proportional to altitude), battery status, and coverage area in nautical miles.

SENSOR FUSION

Multi-sensor track correlation and kill chain visualization. Fuses data from EO/IR, RADAR, SIGINT, SONAR, and other sensors to create a unified tactical picture.

DOCUMENTS

Generate 5-paragraph OPORDs and CONOPs from live mission state. Mission briefing generator. All documents can be downloaded.

INTEGRATIONS

Integration hub for external systems:

  • PX4 SITL — MAVLink bridge for simulated drones
  • TAK/ATAK — Cursor-on-Target XML exchange (UDP/TCP)
  • Link 16 — TADIL J tactical data link simulation
  • ROS 2 — rclpy bridge for robotic systems
  • MQTT / DDS / Kafka — Message bus adapters
  • VMF / STANAG / NFFI / OGC — Military standard bridges
System & Admin

SYSTEM SCRIPTS

Commander access required. Enabled only when AMOS_ENV=dev or AMOS_ALLOW_GIT_UI=true.

Git operations interface for managing the AMOS codebase from the UI.

Status Tab

Shows current branch, modified/staged/untracked files, and diff preview. Refresh to see latest changes.

Commit Tab

  • Select specific files to stage (no blind stage-all)
  • Enter commit message and click COMMIT
  • Optional: check "Push after commit" to auto-push
  • All commits are audited with operator name

Branches Tab

  • View all branches with current branch highlighted
  • Switch branches via the CHECKOUT button
  • Checkout is a danger-zone operation — requires confirmation

Log Tab

Recent commit history with hash, author, date, and message.

Push / Pull

  • Push and Pull are danger-zone operations
  • Both require confirmation modal before executing
  • Only one git operation can run at a time (operation lock)

Security

  • All operations are rate-limited to one at a time via threading lock
  • No arbitrary shell execution — only whitelisted git subcommands
  • Every mutation is logged to the audit trail
  • Disabled in production unless explicitly enabled via environment variable

EDITION DASHBOARD

View and manage AMOS edition features.

Edition Status

Shows current edition (Open or Enterprise), total features, and enabled count.

Feature Bundles

Features are grouped into 5 bundles:

  • Mission Intelligence — Cognitive engine, NLP, COA, commander support, learning, HAL, predictions, wargame, red force, docs
  • Swarm & Autonomy — Swarm intelligence, autonomous behaviors
  • Secure Operations — COMSEC encryption, security audit logging
  • Defense Integration — TAK, Link 16, VMF, STANAG, NFFI, OGC, Kafka bridges
  • Simulation & Effects — Effects chain, ISR/ATR, space, HMT, kill web, EW, SIGINT, cyber, contested

Toggling Features

  • Runtime-safe features can be toggled on/off instantly via the UI
  • Integration features (TAK, Link 16, COMSEC, etc.) require a server restart to change
  • Toggle switches are greyed out for restart-required features
  • All toggles are audited and commander only

The edition itself (Open vs Enterprise) cannot be switched at runtime. Update AMOS_EDITION in .env and restart.

API REFERENCE

All API routes are versioned at /api/v1/. Legacy /api/ routes still work but include deprecation headers.

Key Endpoints

  • GET /api/v1/assets — All assets with position, health, supplies
  • GET /api/v1/threats — Active and neutralized threats
  • GET /api/v1/theater/list — All theater locations
  • POST /api/v1/theater/switch — Switch active theater
  • GET /api/v1/healthz — Liveness probe (no auth)
  • GET /api/v1/readyz — Readiness probe (no auth)
  • POST /api/v1/voice/command — Voice command processing
  • GET /api/v1/edition/status — Edition and feature flags
  • GET /api/v1/scripts/git/status — Git repository status
